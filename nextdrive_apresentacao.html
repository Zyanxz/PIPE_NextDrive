Introdução

NextDrive — Sistema de Locação de Veículos

API REST em Node.js + Express com banco de dados PostgreSQL. O sistema permite cadastro de usuários, listagem e aluguel de carros, controle de locações e um painel administrativo.

Arquitetura e rotas

Todas as rotas passam por middlewares de validação com zod, autenticação JWT e rate limiting. A lógica de negócio está concentrada em server.js.

POST/registrocria usuário, hash bcrypt
POST/loginautentica, retorna JWT
GET/carroslista carros (filtro ?available)
POST/carrosadmin only, upload de imagem
POST/alugartransação + lock pessimista
POST/devolvertransação + lock pessimista
GET/me/locacoeshistórico do usuário logado
GET/usuarios/:idperfil (próprio ou admin)

Middlewares verificarToken e verificarAdmin protegem todas as rotas sensíveis. Pool de conexões do pg gerencia concorrência de banco automaticamente.

Concorrência

Como o sistema lida com acesso simultâneo

O ponto crítico é o aluguel: dois usuários não podem pegar o mesmo carro ao mesmo tempo. O sistema usa duas técnicas combinadas:

Transação ACID

BEGIN / COMMIT / ROLLBACK garantem que todas as operações do aluguel ocorrem juntas ou nenhuma ocorre.

FOR UPDATE (lock pessimista)

A linha do carro é bloqueada no banco enquanto a transação está aberta. Outras conexões aguardam.

// /alugar — trecho central await client.query("BEGIN"); const carro = await client.query( "SELECT * FROM carros WHERE id = $1 FOR UPDATE", [idCarro] ); // ↑ bloqueia a linha — concorrente aguarda aqui if (!carro.rows[0].disponivel) throw new Error("Carro indisponível."); await client.query("UPDATE carros SET disponivel = false ..."); await client.query("COMMIT");

Resultado: mesmo com 10 requisições simultâneas para o mesmo carro, apenas 1 vai ter sucesso. As demais recebem "Carro indisponível." — comportamento correto.

Paralelização

No frontend (index.js), ao carregar a página, as duas listas de carros são buscadas em paralelo — não uma depois da outra:

Falhas identificadas

Problemas reais no código atual

1. Token JWT não tem refresh — expira em 24h sem renovação

O token é gerado uma vez no login com expiresIn: "24h" e não existe rota de refresh. Após 24h, o usuário perde a sessão sem aviso e precisa logar de novo.

2. CPF e CNH não têm validação de formato

O schema Zod aceita qualquer string não vazia para CPF e CNH. Qualquer valor como "abc" ou "00000000000" passa na validação. Não há verificação de dígitos, máscara ou unicidade de CPF.

3. Senha no retorno do registro

No POST /registro, o retorno usa resultado.rows[0] direto — que inclui o campo senha (hash). Deveria usar publicUser() como as outras rotas fazem.

4. Rate limit único para todas as rotas

O rate limit (100 req / 15 min) é global — aplica-se igualmente a GET /carros e POST /login. Rotas de autenticação deveriam ter um limite muito mais agressivo para evitar força bruta.

5. Sem cálculo de valor da locação

A tabela locacoes registra data_inicio e data_fim, mas nenhuma rota calcula o valor cobrado com base em preco_diaria. O campo existe no carro mas nunca é usado no fluxo de devolução.

6. Arquivo arv.env no repositório

O arquivo arv.env está rastreado pelo git e contém credenciais de banco. Mesmo que as senhas sejam de desenvolvimento, esse padrão é perigoso e não deve ir para produção.

Problemas futuros

O que vai quebrar em escala

Pool sem config explícita

O pool usa o padrão do pg (max 10 conexões). Em produção com tráfego real, isso gera fila de espera e timeout.

Server único

O app.listen é single-process. Node.js não usa múltiplos núcleos por padrão — o servidor vai saturar com carga alta.

Uploads locais

Imagens salvas em /uploads no servidor. Em múltiplas instâncias ou containers, os arquivos desaparecem.

Sem paginação

SELECT * FROM carros e SELECT * FROM locacoes sem LIMIT. Com milhares de registros, retorna tudo.

Lock pessimista em escala

O FOR UPDATE resolve race condition mas em alta concorrência gera fila de espera no banco. Com muitos aluguéis simultâneos, a latência sobe porque cada transação bloqueia a linha até o COMMIT.

Soluções

O que fazer para resolver cada problema

1

Token refresh

Adicionar rota POST /refresh que aceita um refresh token de longa duração e retorna novo access token. Access token com expiração curta (15min), refresh com 7 dias.

2

Validar CPF/CNH com biblioteca

Usar cpf-cnpj-validator ou regex com verificação de dígitos. Adicionar constraint UNIQUE no banco para CPF.

3

Rate limit por rota

Rate limit de 5 tentativas / 15min para /login e /registro. Rate limit generoso para rotas de leitura. Usar express-rate-limit por rota em vez de global.

4

Cluster mode / PM2

Rodar pm2 start server.js -i max para usar todos os núcleos da CPU. Em container, usar cluster nativo do Node ou múltiplas réplicas atrás de load balancer.

5

Uploads em S3 / object storage

Substituir Multer local por Multer-S3 ou envio direto para AWS S3 / Cloudflare R2. URLs de imagem passam a ser persistentes e independentes do servidor.

6

Paginação nas listagens

Adicionar parâmetros ?page=1&limit=20 nas rotas de listagem. No banco: LIMIT $1 OFFSET $2. Resposta inclui total para o frontend paginar.
